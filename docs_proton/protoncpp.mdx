# protoncpp

`protoncpp` is the C++ runtime for Proton that:

- Loads a YAML configuration at startup
- Materializes **bundles** and **signals** from that config
- Creates per-peer **connections** that use an underlying **transport** (UDP4 or serial)
- Sends/receives protobuf-encoded `Bundle` messages, updates local bundle state, and invokes callbacks

## Runtime configurability

`protoncpp` is runtime configurable in the sense that the data model and network topology are not compiled into your application:

- The set of nodes, endpoints, connections, bundles, and signals come from a YAML file.
- Signals can be initialized with defaults or constant values in YAML.
- The transport type (UDP4 vs serial) and endpoint parameters are defined in YAML.

Changing the YAML changes what the runtime constructs on next startup. (There is no live reload / dynamic reconfigure API in the current C++ runtime.)

### Node lifecycle

`proton::Node` is a small state machine:

- `PROTON_NODE_UNCONFIGURED` (default)
- `configure()` builds bundles + connections -> `PROTON_NODE_INACTIVE`
- `activate()` starts connection threads (+ heartbeat thread if enabled) -> `PROTON_NODE_ACTIVE`

Construction can auto-run these transitions:

```cpp
proton::Node node("/path/to/config.yaml", "a300", /*auto_configure=*/true, /*auto_activate=*/true);
node.spin();
```

Internally, `configure()`:

- Adds every configured bundle (see "Bundles and signals" below)
- Extracts this node's `NodeConfig` by name
- Scans configured `connections` and creates a `proton::Connection` to each peer
- Creates implicit heartbeat bundles depending on heartbeat settings

## Configuration file format

The YAML is loaded by `proton::Config` (YAML-CPP) and decoded into:

- `nodes[]`: per-node endpoints + heartbeat settings
- `connections[]`: pairs of node endpoints that should communicate
- `bundles[]`: named bundle definitions, each with signals, producers, and consumers

At minimum you need:

- A `nodes` entry for each node name you will run
- One or more `connections` that include that node name
- One or more `bundles`

### Nodes and endpoints

Each node has one or more `endpoints` keyed by an integer `id`. A connection references endpoints by this `id`.

- `type: udp4` requires `ip` and `port`
- `type: serial` requires `device`

Heartbeat is configured per node:

- `heartbeat.enabled` (default `false`)
- `heartbeat.period` in ms (default `1000`)

### Connections

A connection is a pair of `{node, id}` endpoints:

```yaml
connections:
  - first:  {node: a300, id: 0}
    second: {node: j100, id: 0}
```

During `configure()`, each connection involving the local node becomes a `Connection` to the peer.

Implementation constraints:

- The two endpoints in a connection must have the same transport `type` (otherwise `Connection` throws).
- The endpoint `id` must exist in each node's `endpoints` map.

### Bundles

Bundles are declared with:

- `name`: human-readable bundle name
- `id`: numeric bundle ID (must be non-zero; `0` is reserved for heartbeat)
- `producers`: node name(s) allowed to publish the bundle
- `consumers`: node name(s) that receive the bundle
- `signals`: list of signal definitions

Producers/consumers can be either a scalar string or a YAML sequence.

## Bundles and signals

A **bundle** is a protobuf message (`proton::Bundle`) with a numeric `id` and a repeated list of `Signal` messages (`signals[]`).

Bundles are managed through `proton::BundleHandle`:

- Holds a `std::shared_ptr<Bundle>` (the live data)
- Owns a map of `SignalHandle`s for named access and typed get/set
- Stores producers/consumers and an optional callback

### Signals: types, length, capacity

Signals are represented by a protobuf oneof (`proton::Signal`) and wrapped by `proton::SignalHandle`.

Supported YAML `type` strings include scalars and lists:

- Scalars: `double`, `float`, `int32`, `int64`, `uint32`, `uint64`, `bool`, `string`, `bytes`
- Lists: `list_double`, `list_float`, `list_int32`, `list_int64`, `list_uint32`, `list_uint64`, `list_bool`, `list_string`, `list_bytes`

Two sizing concepts are used:

- `length`: number of list elements (for list types)
- `capacity`: maximum size for `string`/`bytes` element storage (and per-element capacity for `list_bytes`)

Rules enforced by config parsing:

- List types require `length` unless a constant `value` is provided.
- `string`, `bytes`, `list_string`, `list_bytes` require `capacity` unless a constant `value` is provided (in which case it is derived).

### Constant vs non-constant signals (important)

Signals become **constant** if YAML defines a `value`.

Implementation detail with real impact:

- Non-constant signals are appended into the protobuf bundle's `signals[]` and therefore are transmitted.
- Constant signals are constructed as standalone `Signal` objects and are not added to the bundle's `signals[]`; they exist as local, read-only configuration values.

Attempting to `setValue()` on a constant signal throws at runtime.

### Example bundle YAML

```yaml
bundles:
  - name: status
    id: 1
    producers: a300
    consumers: [j100]
    signals:
      - {name: mode, type: uint32}
      - {name: nameplate, type: string, capacity: 32}
      - {name: temperatures, type: list_float, length: 8}
      # Constant local-only value (not transmitted)
      - {name: model, type: string, value: "A300"}
```

### Typed access API

Signals are read/written through templates:

```cpp
auto& status = node.getBundle("status");

status.getSignal("mode").setValue<uint32_t>(3);
uint32_t mode = status.getSignal("mode").getValue<uint32_t>();

// List element update
status.getSignal("temperatures").setValue<float>(0, 42.0f);

// bytes: set entire buffer or a single byte
status.getSignal("payload").setValue<proton::bytes>({0x01, 0x02});
status.getSignal("payload").setValue<uint8_t>(0, 0xFF);

// list_bytes: set element, or element byte at subindex
status.getSignal("lights").setValue<proton::bytes>(0, {0x10, 0x20, 0x30});
status.getSignal("lights").setValue<uint8_t>(0, 2, 0x7F);
```

Out-of-range indices throw `std::out_of_range`.

## Sending and receiving bundles

### Sending

`Node::sendBundle()` serializes the bundle protobuf to a byte buffer and writes it to each configured consumer connection that is currently connected.

Transport-specific behavior:

- UDP4: the protobuf bytes are sent as a single datagram
- Serial: the protobuf bytes are wrapped in a Proton serial frame (see "Serial transport" below)

### Receiving, updating, callbacks

Each `Connection` runs in its own thread and polls the transport for the next message. When it parses a `Bundle`, it invokes the node's `readCompleteCallback`, which pushes it into a thread-safe queue (`SafeQueue`).

The application thread typically calls `spinOnce()` or `spin()`:

- Blocks waiting for the next received bundle
- Finds the matching local `BundleHandle` by bundle ID (or heartbeat producer)
- Updates the `Bundle` stored in that handle
- Increments RX counters and invokes the handle callback if registered

Callback registration is guarded by configuration:

- `registerCallback(bundle, cb)` only succeeds if the current node is listed as a consumer of that bundle.
- `registerHeartbeatCallback(producer, cb)` only succeeds if the current node is a consumer of that producer's heartbeat.

## Heartbeats

Heartbeats are implicit bundles created at runtime (not declared in the `bundles:` section).

- Heartbeat bundle ID is always `0` (reserved)
- Bundle name is the producer node name
- It contains a single `uint32` signal named `heartbeat`

If this node's heartbeat is enabled:

- `activate()` starts a heartbeat thread that periodically increments the `heartbeat` signal and sends the heartbeat bundle to all peers.

If a peer's heartbeat is enabled:

- The `Connection` starts a watchdog thread that marks the peer inactive if heartbeats are late.
- Any received bundle with `id == 0` triggers `Connection::heartbeat()` (updates last heartbeat timestamp and sets peer state active).

## Transport architecture

`protoncpp` uses a small abstraction:

- `Transport`: connect/disconnect/read/write + transport state
- `TransportManager`: owns a `std::unique_ptr<Transport>` and wraps calls to update state and accumulate RX/TX byte counters

Each `Connection` is a `TransportManager` and chooses its concrete transport based on endpoint `type` in the YAML.

### UDP4 transport

`Udp4Transport` uses two sockets:

- A "node" socket that is `bind()`-ed to the local endpoint and used for `recv()`.
- A "peer" socket that is `connect()`-ed to the peer endpoint and used for `send()`.

Implementation notes:

- The peer socket is set non-blocking; the node socket is created blocking.
- `read()` blocks in `recv()` until a datagram arrives.
- There is no extra framing: the UDP payload is the protobuf bytes.

### Serial transport

`SerialTransport` uses ASIO's `serial_port` and frames the protobuf payload.

Write path:

- Builds a framed packet: header (`0x50 0x52` + length) + payload + CRC16
- Writes the full frame to the serial port

Read path:

- Synchronizes to frame start by scanning for `0x50`
- Reads the remaining header and extracts payload length
- Reads payload + CRC16
- Validates the CRC16

In `Connection::pollForBundle()`, serial reads subtract the serial frame overhead before parsing protobuf.

## Threading model and stats

At runtime you typically have:

- 1 connection thread per peer (`Connection::spin()`)
- Optional 1 heartbeat watchdog thread per peer (if peer heartbeat enabled)
- Optional 1 node heartbeat sender thread (if local heartbeat enabled)
- Optional 1 stats thread (`Node::startStatsThread()`)

`startStatsThread()` computes per-bundle RX/TX counts per second and `printStats()` displays:

- Connection transport state and KB/s rates
- Produced/consumed bundle rates
- Heartbeat rates

## Notes / current quirks

- Bundle IDs cannot be `0` in YAML; `0` is reserved for heartbeat bundles.
- Constant signals (YAML `value:`) are local-only and not serialized into outgoing bundles.
- `Node::spin()` is an infinite loop; you generally run it on the main thread or in your own thread.
